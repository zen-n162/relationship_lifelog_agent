Local-first relationship memory review

Relationship Lifelog Agent

Relationship Lifelog Agentは、既存のローカルlifelogアプリを上流の記憶ツールとして扱い、 関係にまつわる記録を慎重に整理するためのローカル専用アプリケーションです。 関係を断定・診断するAIではなく、観測できる根拠を集めて見直すためのレビュー支援ツールです。

Local only Mock adapters first SQLite Gradio UI Manual profiles Safety evals No external APIs share=False

このアプリがすること

質問を意図に分類

日本語の質問をルールベースで分類します。MVPではLLM分類を使わず、 「喧嘩はどのくらいしている？」などを決定的にintentへ振り分けます。

根拠を集める

標準ではmock adapterを使い、必要に応じてupstream_readonlyで LINE候補、外出候補、メモ候補、月次振り返り候補をread-only取得します。 上流アプリのDBは更新しません。

対象profileを手動設定

relationship label、person source ID、LINE speaker source IDはユーザーが CLIまたはUI設定で手動選択します。AIは恋人ラベルやID対応を推定しません。

dry-runで候補抽出

実データまたはmock dataからrelationship event candidatesを抽出し、 Markdown reportだけを生成します。reportはcounts-only、 redacted、privateのprivacy levelを選べます。 dry-runではrelationship DBへ候補を書き込みません。

慎重に回答する

件数、日付、候補の集計はPythonで計算します。喧嘩や仲直りを断定せず、 事実と推定を分けて、根拠・信頼度・注意を含む回答を返します。

evalで継続確認

mock/private/publicの質問ケースを使い、intent、回答形式、根拠、 禁止表現、public redaction、raw data leakageを継続的に検査します。

doctorで事前診断

実データ運用前にconfig、relationship DB、上流DBのread-only接続、 adapter backend、privacy設定、eval実行可否をredacted出力で確認します。

schema mappingを確認

上流SQLite schemaとadapter contractの対応を、table名、column名、 row count、mapping statusだけで診断します。実データ本文は出しません。

counts-only smoke

実upstream_readonly接続でadapterをE2E確認し、LINE/media/event/note/thoughtの 件数、期間coverage、warningだけをreportします。relationship DBへは書き込みません。

backupとrollback

analyze dry-run --writeの前にrelationship DBだけをbackupし、 duplicate、raw leakage、禁止表現を確認します。restoreは明示CLIでのみ実行します。

review結果を優先

verifiedとcorrectedはAI候補より優先し、 rejectedや冗談扱いのeventは通常回答から除外します。 回答には人間確認済み件数、AI推定のみ件数、除外済み件数、要再分析件数を含めます。

MVPで答えられる質問

現在のMVPは、mock dataまたはopt-inのread-only上流adapterで次のような質問に対応します。

  • 喧嘩はどのくらいしている？
  • 喧嘩はいつ多かった？
  • 喧嘩の後どこへ行っている？
  • その時、自分は何を考えていた？
  • 2025年1月の関係を振り返って
回答は「喧嘩候補」「外出候補」「反省/不安候補」のように候補として表現します。 相手の気持ち、関係ラベル、親密さ、診断的な結論は断定しません。 profile未設定時は、具体的な関係対象を前提にせず、手動設定の案内を返します。

処理の流れ

1
Chat UI

ユーザーがGradioのシンプルなチャット画面で質問します。設定Accordionでbackend、profile、date range、post-conflict window、private/public modeを選べます。

2
Router

質問文からintentを抽出します。複数intentも許可します。

3
Planner / Executor

intentに応じて必要なadapter呼び出しを決め、設定されたbackendとdate rangeから根拠を集めます。relationship DBに保存済みcandidateがあれば、それも候補として読みます。

4
Analytics / Answer composer

件数や日付を決定的に集計し、要約、集計、具体例、根拠、信頼度、注意を含む日本語回答を作ります。

5
Privacy guard

回答後に禁止表現やpublic modeで出してはいけない情報を検査・匿名化します。

6
Review action

回答にrelationship DB由来のcandidateが含まれる場合、最小Review accordionからverify、reject、mark_as_misunderstandingなどを保存できます。上流データや元sourceは変更しません。

主要コンポーネント

領域 役割 現在の状態
adapters/ 上流lifelogアプリへの読み取り境界を定義します。 mock adapterを標準にしつつ、upstream_readonlyでSQLite read-only接続をopt-in利用できます。未設定時は安全なwarningと空結果を返します。
agent/ 質問分類、計画、実行、回答生成を担当します。 ルールベースrouter、date range付きadapter計画、relationship DB保存済みcandidateの読み込み、決定的回答生成を実装済みです。
analytics/ 喧嘩候補、外出候補、月次レビューなどを集計します。 Pythonで件数・日付・根拠強度を扱い、dry-run候補抽出reportを生成します。
db/ relationship DBのschemaとrepository層を提供します。 SQLite schema、migration、基本CRUD、手動profile管理を実装済みです。
privacy/ 禁止表現検出、public mode匿名化、安全な回答への変換を行います。 person name、relationship label、GPS、path、raw excerptの保護をテスト済みです。
review/ ユーザーによるcandidate review actionを保存します。 relationship_review_actionsに履歴を残し、relationship_events.review_statusを更新します。reject済みcandidateは通常回答から除外します。
eval/ mock / upstream_readonly / private / publicの安全性と回答品質を継続評価します。 python eval/run_eval.pyでMarkdownまたはJSONの評価結果を出力します。upstream_readonly caseはローカル設定済みの場合だけ任意実行します。
doctor config.local.yaml、relationship DB、上流DB、privacy設定を安全に診断します。 python -m relationship_lifelog_agent.cli doctorでOK/WARN/ERRORを表示します。出力はcounts-onlyかつprivate pathをredactします。
upstream inspect 上流SQLite schemaとadapter method coverageを診断します。 python -m relationship_lifelog_agent.cli upstream inspectでMarkdownまたはJSONを出力します。生LINE本文、生メモ本文、正確GPS、顔情報、写真パスは出力しません。
upstream smoke 実upstream_readonly adapterをcounts-onlyでE2E確認します。 source counts、date range coverage、warnings、redaction status、write_count: 0を出します。relationship DBには書き込みません。
upstream identities 手動profile作成用のperson / LINE speaker ID候補を安全に確認します。 redacted modeでは人物名や表示名を匿名化し、LINE本文、メモ本文、正確GPS、顔情報、写真パス、private path、AI推定の関係ラベルを出しません。AIによるpersonとLINE speakerの自動リンクも行いません。
ui/ ChatGPT風の最小Gradio UIを構成します。 チャット履歴、入力欄、送信ボタン、backend/profile/date range/window/mode/debug、最小review actionを含む折りたたみ設定を実装済みです。

データと安全境界

  • 既存のpersonal_lifelog_ragとnotes_lifelog_ragは上流ツールとして扱い、破壊的に編集しません。
  • relationship DBには、関係固有の派生情報とsource pointerを保存し、上流のraw private dataを丸ごとコピーしません。
  • relationship_profilesは手動設定のみを保存し、label_sourceはuser_manual固定です。
  • 外部API、cloud upload、モデル自動ダウンロードは使いません。
  • Gradioは127.0.0.1で起動し、share=Trueは禁止です。
  • public modeでは実人物名、恋人ラベル、LINE全文、メモ全文、正確GPS、顔crop、実写真、private pathを出しません。
重要な設計原則: relationship judgment AIではなく、relationship evidence review AIとして振る舞います。

SQLite schema

relationship DBは次のテーブルを持ちます。

  • relationship_profiles: 手動設定された対象profile
  • relationship_events: 喧嘩候補、仲直り候補、外出候補、感情メモ候補など
  • relationship_event_evidence: source pointerと短い根拠情報
  • interaction_metrics: メッセージ数、返信遅延、沈黙期間などの観測指標
  • post_conflict_activities: 喧嘩候補後の外出・活動候補
  • relationship_review_actions: verify、reject、correctなどの人間レビュー操作

起動とテスト

conda環境で依存関係を入れたあと、ローカルUIを起動できます。

conda activate relationship_lifelog_agent
python -m relationship_lifelog_agent.app

標準のURLはhttp://127.0.0.1:7862です。

起動時だけ別のローカルポートを使う場合は、--portを指定します。hostは127.0.0.1のまま、share=Trueは無効のままです。

python -m relationship_lifelog_agent.app --port 7863

実データ運用前にはdoctorで安全設定とread-only接続を確認できます。

python -m relationship_lifelog_agent.cli doctor
python -m relationship_lifelog_agent.cli doctor --backend upstream_readonly
python -m relationship_lifelog_agent.cli doctor --format json

上流schemaとadapter mappingの対応はinspectで確認できます。

python -m relationship_lifelog_agent.cli upstream inspect \
  --backend upstream_readonly \
  --format markdown \
  --output data/exports/upstream_schema_inspection.md

adapterのE2E接続はcounts-only smokeで確認できます。

python -m relationship_lifelog_agent.cli upstream smoke \
  --backend upstream_readonly \
  --date-from 2025-01-01 \
  --date-to 2025-01-31 \
  --profile-id 1 \
  --output data/exports/upstream_smoke_2025_01.md

手動profile作成のID候補はidentity inventoryで確認できます。redacted modeでは名前や表示名を匿名化します。

python -m relationship_lifelog_agent.cli upstream identities \
  --backend upstream_readonly \
  --kind all \
  --privacy-level redacted \
  --format markdown \
  --output data/exports/upstream_identities.md

profileはCLIから手動作成できます。

python -m relationship_lifelog_agent.cli profile create \
  --profile-name "Aさん" \
  --person-source-id "plr:person:4" \
  --line-speaker-source-id "plr:line_speaker:2" \
  --relationship-label "partner"

relationship event candidatesはdry-runでMarkdown reportとして生成できます。標準ではDBに候補を書き込みません。

reportの--privacy-levelは三段階です。 counts-onlyは件数・期間・根拠強度分布だけを出し、本文excerptやsource pointerを出しません。 redactedは匿名化summaryとredacted source pointerだけを出します。 privateは短いexcerptを許可しますが、正確GPS、顔情報、写真path、private path、断定表現は出しません。 public modeではprivate levelを拒否します。 

python -m relationship_lifelog_agent.cli analyze dry-run \
  --profile-id 1 \
  --date-from 2025-01-01 \
  --date-to 2025-03-31 \
  --backend mock \
  --privacy-level counts-only \
  --output data/exports/dry_run_relationship_2025Q1.md

--writeを明示した場合だけ、relationship DBへcandidateとして保存します。保存するのはevent summary、source pointer、privacy levelに応じた短いexcerpt、confidence、evidence strengthで、生LINE全文、生メモ全文、正確GPS、顔情報、写真実体は保存しません。counts-onlyとredactedではexcerptを保存しません。保存時のstatusはcandidate、review_statusはunreviewedです。

--writeの直前にはdata/backups/へrelationship DB backupを作成し、 candidate count、duplicate count、raw leakage scan、禁止表現scan、profile/date range/privacy levelを確認します。 backup pathはCLI出力ではredactします。 

python -m relationship_lifelog_agent.cli analyze dry-run \
  --profile-id 1 \
  --date-from 2025-01-01 \
  --date-to 2025-03-31 \
  --backend mock \
  --privacy-level private \
  --write

backupとrestoreはrelationship DBだけを対象にした明示コマンドです。上流DBは変更しません。

python -m relationship_lifelog_agent.cli db backup
python -m relationship_lifelog_agent.cli db backups
python -m relationship_lifelog_agent.cli db restore \
  --backup-path data/backups/relationship_YYYYMMDD_HHMMSS.sqlite

回答ではreview済みrelationship eventsをAI推定候補より優先します。 verifiedとcorrectedは人間確認済みとして扱い、 needs_reanalysisは注意付き候補として残し、 rejectedと冗談扱いのhidden eventは通常回答から除外します。 mark_as_misunderstandingは軽いすれ違い、 mark_as_reconciledは仲直り候補として集計します。

継続評価はmock backendで安定実行できます。public/private安全性、禁止表現、根拠の有無、弱い根拠への注意、関係ラベル推定の抑制を確認します。

python eval/run_eval.py
python eval/run_eval.py --format json --output data/exports/eval_result.json
pytest

今後の拡張方針

  • 上流アプリ接続はadapter境界の内側に限定し、SQLiteはmode=roのread-only接続で扱います。
  • adapter evidenceはsource_pointer、source_type、sensitivity、confidence、evidence_strengthを持ちます。
  • LLMを使う場合もlocal LLMをユーザー設定でのみ利用し、分類補助や要約補助に限定します。
  • 顔identity、LINE speaker identity、relationship labelはAIで自動推定しません。
  • 仕様、intent、DB schema、安全方針、UI挙動が変わったら、このHTML説明も更新します。
